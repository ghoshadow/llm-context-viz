# Codex Calibration Constants Design

## Purpose

Codex JSONL sessions do not always contain the full model request. In the captured run on 2026-06-26, the JSONL preserved `session_meta.base_instructions`, developer/user `response_item`s, model output items, and `token_count` usage. It did not preserve the full `tools` array from the `/responses` request. Because the turn inspector uses category composition to explain context growth, Codex needs a dedicated calibration path that can fill missing core categories from an API capture without overwriting real per-request usage.

This feature now uses the shared normalized calibration schema for Claude Code and Codex. Codex-specific capture logic still lives in a Codex extractor/launcher, but storage, API routes, UI rendering, and pipeline fallback behavior are source-aware and schema-compatible with future agent adapters such as opencode and openclaw.

## Goals

- Automatically extract Codex core context constants from a controlled `/responses` request.
- Use those constants to fill missing Codex `comp` categories, especially `tool_defs`.
- Keep `cumTotal`, `maxInput`, cache hit, and output totals backed by real JSONL `token_count` usage whenever present.
- Preserve the existing JSONL-only behavior when no Codex calibration file exists.
- Make the calibration source visible in the UI so users can tell whether a category came from JSONL or capture calibration.
- Persist the raw intercepted API log locally so future parser changes can be audited against the original wire payload.

## Non-Goals

- Do not replay requests or reconstruct every token exactly.
- Do not store API keys or authorization headers.
- Do not require Codex users to run calibration before importing logs.
- Do not replace actual `token_count.input_tokens` with estimates.
- Do not require users to manually upload API logs for the normal Codex calibration path.

## Inputs

### Automatic Codex Capture

The primary Codex calibration path is automatic:

1. Server reads the active Codex model provider and base URL from `~/.codex/config.toml`.
2. Server starts a temporary local HTTP reverse proxy on `127.0.0.1:<free-port>`.
3. Server launches a one-shot calibration command:

   ```bash
   codex exec --ephemeral --json \
     -c 'model_providers.OpenAI.base_url="http://127.0.0.1:<free-port>"' \
     -s read-only \
     -C <workspace> \
     'Calibration probe: reply with "ok".'
   ```

4. The proxy forwards to the original configured base URL and captures the first `/responses` request and response.
5. The extractor reads the captured request body and streamed response usage.
6. The raw capture is written to `.codex-trace/codex-api-log-*.jsonl` for audit and future parser improvements.

This approach avoids HTTPS MITM and certificate installation because the child Codex process is explicitly pointed at the local proxy. The proxy then talks to the original upstream using the original scheme (`http` or `https`).

### Raw Codex API Capture

The automatic path persists the Codex capture log as JSONL generated by the local HTTP reverse proxy. Developers can reuse this raw file for offline parser debugging, but the normal UI workflow does not require users to upload API logs manually. Each line has the same high-level shape as the existing proxy logs:

```json
{
  "request": {
    "method": "POST",
    "upstream_url": "http://.../responses",
    "headers": { "authorization": "[REDACTED]" },
    "body": {}
  },
  "response": {
    "status_code": 200,
    "headers": {},
    "body": "event: response.created\n..."
  }
}
```

The extractor should accept entries whose request URL path is `/responses` and whose request body has `model`, `instructions`, `input`, and `tools`.

### Codex JSONL

The Codex session JSONL remains the primary per-session source. It may contain:

- `session_meta.base_instructions.text`
- `session_meta.dynamic_tools` in some older or different logs
- developer `response_item` blocks
- user environment and prompt blocks
- reasoning, tool call, tool output, assistant message output items
- `event_msg.token_count.info.last_token_usage`

## Extracted Constants

Constants are written as source-aware project files under the local trace directory:

- Claude: `.claude-trace/system-constants.json`
- Codex: `.codex-trace/codex-system-constants.json`
- Future adapters: `.<source>-trace/<source>-system-constants.json`

Recommended normalized schema:

```json
{
  "schemaVersion": 1,
  "source": "codex",
  "constantsSource": "project",
  "path": "/workspace/.codex-trace/codex-system-constants.json",
  "cwd": "/workspace",
  "wireApi": "responses",
  "appliedAt": "2026-06-26T00:00:00.000Z",
  "cliVersion": "0.142.2",
  "model": "gpt-5.5",
  "rawLogPath": "/workspace/.codex-trace/codex-api-log-2026-06-26T00-00-00-000Z.jsonl",
  "categories": {
    "sysPrompt": { "chars": 21335, "detailKey": "codex.instructions", "origin": "capture" },
    "tool_defs": { "chars": 12345, "detailKey": "codex.tools", "origin": "capture" },
    "skills": { "chars": 15644, "detailKey": "codex.skills", "origin": "capture" },
    "mcp": { "chars": 1014, "detailKey": "codex.plugins", "origin": "capture" },
    "reminders": { "chars": 341, "detailKey": "codex.runtime", "origin": "capture" }
  },
  "usage": {
    "firstRequestInputTokens": 13749,
    "firstRequestCachedTokens": 9600,
    "firstRequestOutputTokens": 360,
    "firstRequestReasoningTokens": 270
  },
  "toolNames": ["exec_command"],
  "hashes": {
    "instructions": "sha256...",
    "tools": "sha256..."
  },
  "details": {
    "codex.instructions": "# codex.instructions\n\n...",
    "codex.tools": "# codex.tools\n\n```json\n[]\n```"
  }
}
```

`categories.tool_defs.chars` must be the character length of `JSON.stringify(body.tools)`. `categories.sysPrompt.chars` must be `body.instructions.length`.

Developer input classification:

- Blocks containing `<skills_instructions>` go to `categories.skills`.
- Blocks containing `<plugins_instructions>` go to `categories.mcp`.
- Other developer blocks, including `<permissions instructions>`, `<app-context>`, and `<collaboration_mode>`, go to `categories.reminders`.

For the current category model, Codex plugin instructions map to the existing `mcp` category. A future UI cleanup can introduce a separate `plugins` category.

## Pipeline Behavior

Load Codex constants through the shared source-aware calibration service and pass the resulting `NormalizedCalibration` into `runCodexPipeline`.

When assembling Codex core composition:

1. `sysPrompt`
   - Prefer `session_meta.base_instructions.text`.
   - Fall back to normalized `categories.sysPrompt.chars`.

2. `tool_defs`
   - Prefer `session_meta.dynamic_tools` if present.
   - Fall back to normalized `categories.tool_defs.chars`.

3. `skills`
   - Prefer developer JSONL blocks matching `<skills_instructions>`.
   - Fall back to normalized `categories.skills.chars`.

4. `mcp`
   - Prefer developer JSONL blocks matching `<plugins_instructions>`.
   - Fall back to normalized `categories.mcp.chars`.

5. `reminders`
   - Prefer developer JSONL runtime blocks.
   - Fall back to normalized `categories.reminders.chars`.

6. `memory`
   - Remain `0` for Codex unless a future Codex JSONL field gives explicit memory file content.

7. `cumTotal`, `maxInput`, `maxCacheHit`, `outTok`
   - Continue to use JSONL `token_count` values. Calibration constants only fill composition categories.

All fallback values should be converted using the existing token estimator unless the calibration file later stores explicit per-category tokens.

## API Design

Extend the calibration route without breaking existing Claude clients.

- `POST /api/calibrate/auto/start`
  - Start a temporary local reverse proxy.
  - Accept `{ source: "codex", cwd, prompt?, targetHost?, timeoutMs? }`.
  - Spawn `codex exec --ephemeral` with a temporary `base_url` override.
  - Capture the first `/responses` request/response.
  - Persist the raw capture under `.codex-trace/` with sensitive headers redacted.
  - Extract constants.
  - Return a pollable job snapshot whose `result` is a `NormalizedCalibration`.

- `GET /api/calibrate/auto/:jobId`
  - Poll automatic capture/extraction status.
  - Return `logFile`, progress output, and a normalized result when ready.

- `PUT /api/calibrate/apply`
  - Accept `{ source: "codex", cwd, summary, details, cliVersion, model, wireApi, rawLogPath }`.
  - Write `.codex-trace/codex-system-constants.json`.
  - Existing Claude calls without `source` continue writing `.claude-trace/system-constants.json`.

- `GET /api/calibrate/current?source=codex`
  - Return the current Codex constants file if present.
  - Otherwise return a normalized defaults object with empty Codex categories and a "not calibrated" note.

The extraction function should live beside the existing extractor, for example:

`src/pipeline/extract-codex-constants.ts`

It should parse both standard JSON bodies and streamed SSE response bodies to extract first-request usage.

The shared calibration job service and proxy launcher are source-aware. Responsibilities:

- Read `~/.codex/config.toml` and locate the active model provider base URL.
- Allocate a local port and start the reverse proxy.
- Redact sensitive headers before writing the raw capture log.
- Write each captured request/response pair to `.codex-trace/codex-api-log-<timestamp>.jsonl`.
- Spawn the Codex calibration command with a temporary `base_url` override.
- Stop after the child exits or after a timeout.
- Return the raw log path to the extractor.

Timeout behavior:

- If no `/responses` request is seen within 30 seconds, return a clear error.
- If Codex exits non-zero before a request is captured, return the stderr/stdout tail.
- If the upstream request fails, return the upstream error and keep the raw log path/output tail available for debugging when present.

## UI Design

Update `CalibratePage` with a compact source selector:

- Claude Code
- Codex

Claude mode keeps the current instructions and result cards.

Codex mode shows:

- Primary action: `自动校准 Codex 常量`.
- Status output for the automatic flow: proxy started, Codex probe started, request captured, raw log saved, constants extracted.
- Raw log path after success, for example `.codex-trace/codex-api-log-2026-06-26T...jsonl`.
- Extracted cards rendered from normalized categories:
  - Codex CLI version if available from request metadata, otherwise `unknown`
  - Model
  - `sysPrompt`, `tool_defs`, `skills`, `mcp`, and `reminders` chars and estimated tokens when present
  - First request input/cached/output/reasoning tokens

The page should state that Codex JSONL may omit tool definitions and that capture calibration fills that gap.

Automatic calibration uses a staged workflow: capture and extraction first, then `应用常量` writes the normalized constants file using the same result model.

## Trace Safety

The automatic reverse proxy must persist raw API request/response bodies for auditability. Before writing logs it must redact sensitive headers:

- `authorization`
- `cookie`
- keys containing `key`, `token`, `secret`, or `auth`

The raw log may contain model-visible prompt content, tool definitions, tool outputs, reasoning summaries, and encrypted reasoning blobs. This is expected and useful for calibration debugging. It must stay local.

The constants file persists counts, details for inspection, tool names, hashes, model/version metadata, timestamps, and the raw log path. The full request/response body remains in the raw `.codex-trace/codex-api-log-*.jsonl` file rather than being duplicated into the constants file.

The `.codex-trace/` directory remains local-only and must not be committed.

## Testing

Add unit tests for:

- Source-aware proxy launcher behavior for Claude and Codex without spawning real CLIs.
- Extracting Codex constants from a captured `/responses` log.
- Parsing SSE `response.completed` usage.
- Rejecting non-Codex or malformed capture logs.
- Applying Codex constants to `tool_defs` when JSONL lacks `dynamic_tools`.
- Preserving JSONL `token_count` as the source of `cumTotal` and `maxInput`.
- Keeping existing Claude calibration compatibility tests green.

Add one integration-style pipeline test:

- JSONL has `base_instructions` and developer blocks but no `dynamic_tools`.
- Codex constants file has `categories.tool_defs.chars`.
- Resulting `turn.comp.tool_defs` is non-zero.
- `turn.cumTotal` equals JSONL `input_tokens`.

Add one route-level test or service test:

- `POST /api/calibrate/auto/start` returns a source-aware job and extracts constants from a mocked capture log.
- `PUT /api/calibrate/apply` writes `.codex-trace/codex-system-constants.json`.

## Migration

Existing imported sessions will not automatically change until refreshed or re-imported. This is acceptable. The UI can continue to show old values until the user refreshes a session after applying Codex constants.

No database migration is needed because `comp_json` already stores arbitrary category keys and values.

## Open Questions

- Whether to add a first-class `plugins` category later instead of mapping plugins to `mcp`.
- Whether to store explicit token counts per category if the API or tokenizer becomes available.
- Whether automatic calibration should auto-apply by default or require one confirmation click after extraction.
